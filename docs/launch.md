# Launch and community packet

This packet turns the public launch steps into reproducible repo artifacts. It is
offline by design: maintainers can copy the text below into Show HN, Reddit,
project issues, blog posts, or a talk CFP, but the proof claims are always
grounded in commands that run in this repository.

## Proof commands

Run these before any public post:

```bash
python3 scripts/validate_launch_readiness.py
python3 scripts/build_readme_demo.py
python3 -m src.ub_oracle.cli \
  --units examples/readme_demo_units.json \
  --format text \
  --color never \
  --fail-on unknown
make c2rust-corpus-check
make soundness-check
```

The launch validator fails unless the README demo produces both:

- a confirmed `DIVERGENT` C->Rust signed-overflow witness with real
  `clang`/UBSan and `rustc`; and
- a `NO-DIVERGENCE` safe control discharged by the interval pre-pass.

## Show HN / r/rust / r/programming copy

**Title:** Cross-lang-verifier catches translation bugs fuzzing misses

**Short post:**

I built `cross-lang-verifier`, a sound-for-divergence oracle for translated code.
The flagship demo catches a C->Rust signed-overflow bug that differential tests
can miss: C reaches undefined behavior on the witness, while the Rust port is
defined and deterministic. The repo ships the reproducer, the verifier, the
counterexample certificate, SARIF/CI integration, and a c2rust-output corpus.

Reproduce the main claim:

```bash
python3 -m pip install -e .
python3 -m src.ub_oracle.cli \
  --units examples/readme_demo_units.json \
  --format text \
  --color never \
  --fail-on unknown
```

What makes the project different: it does not claim whole-program equivalence
outside its fragment. Unsupported constructs become loud `UNKNOWN`,
`CANDIDATE`, or `NOT-COVERED` verdicts; confirmed divergences are re-executed
against real compilers and carry hash-bound evidence.

## c2rust / TRACTOR / transpiler outreach

Use this as an issue, email, or PR opener for migration/transpiler communities:

> I am testing translated C with `cross-lang-verifier`, which finds
> language-semantics divergences that can survive compilation and fuzzing. The
> repo already includes a c2rust-output corpus and a manifest-driven CLI. I can
> run it on a small translated slice of your project and send only confirmed
> divergences, each with a minimized counterexample and the exact compiler
> evidence. The first pass can be read-only and CI-safe via SARIF.

Suggested first collaboration:

1. Pick one translated module and add a manifest unit.
2. Run `cross-lang-verify --units units.json --sarif findings.sarif`.
3. Triage only confirmed `DIVERGENT` findings; keep `CANDIDATE`/`UNKNOWN`
   results visible but non-blocking.
4. Promote each accepted finding into a regression unit.

## Blog series

1. **The divergence catalogue:** explain why C UB, implementation-defined
   behavior, target panics, and runtime contracts are cross-language hazards.
   Anchor examples: signed overflow, `memcpy` overlap, strict aliasing,
   sequence points, and ABI layout.
2. **A real compiler-backed bug:** walk through the README demo and the c2rust
   `nginx_rate` video path. Include the exact commands, counterexample, and
   sanitizer/toolchain output.
3. **Why fuzzing misses this class:** compare a sanitizer/differential baseline
   against the oracle's semantic witness.
4. **Adding a plugin in a weekend:** start from
   `examples/plugins/float_cast_overflow_oracle.py`, run the SDK conformance
   path, and add a target-semantics pack.

Each post should link to a checked command or test, not a hand-written number.

## Conference talk and recorded demo runbook

**Talk thesis:** translation validation must compare source and target language
semantics, not just object-level behavior. C->Rust UB is the clean wedge; the
same plugin architecture scales to Go, Swift, C++, OCaml, Zig, WebAssembly, and
reverse Rust->C hazards.

**Live demo:**

1. Show the README GIF and explain the witness.
2. Run the CLI command from "Proof commands".
3. Open the SARIF output in code scanning or VS Code.
4. Show `docs/SOUNDNESS_COMPENDIUM.md` to map the finding to a soundness claim.
5. Show `docs/SDK.md` and the external plugin example as the contribution path.

**Backup demo:** use `docs/assets/demo_video.mp4`, regenerated by
`python3 scripts/build_demo_video.py`.

## Contribution ramp

- **Good-first issues:** label small corpus additions, safe negative controls, and
  docs-with-repro-command fixes as the default first contributions.
- **Bug reports:** use the "Divergence report" issue form and include a manifest,
  compiler versions, and whether the finding is confirmed or candidate-only.
- **New oracle/plugin:** start from the plugin request issue form, then implement
  the SPI contract and add a positive witness plus a safe control.
- **PR checks:** run `python3 scripts/validate_launch_readiness.py` when touching
  launch-facing claims, README demo assets, issue templates, or contribution
  paths.

External posting, community replies, conference submission, and DOI minting are
operational tasks outside the repository. This file is the reproducible packet
that makes those actions safe to perform without inventing unverified claims.
