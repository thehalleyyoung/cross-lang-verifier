%% ABSTRACT
%% ======================================================================
\begin{abstract}
C-to-Rust migration mandates from CISA and DoD are accelerating, but
verifying that translated code preserves the semantics of the original
remains unsolved.  The most dangerous translation bugs---signed overflow,
shift~UB, division edge cases, out-of-range casts---are \emph{provably
invisible} to every existing verification approach: they produce
identical LLVM~IR (100\% erasure rate on our benchmark), identical
runtime outputs on~x86, and pass every differential test suite.
We present \textsc{SemRec}, the first and only C$\to$Rust verifier
that detects these IR-invisible UB divergences.  \textsc{SemRec}
encodes C11 and Rust semantics \emph{separately} via a
\emph{$\sigma$-bridge product program}, then discharges the combined
formula with Z3 QF\_BV solving.  On a benchmark of 30 IR-invisible
UB divergences across 6 classes (overflow, shift, division, cast,
negate, compound) plus 10 equivalent negative controls,
\textsc{SemRec} achieves \textbf{100\% detection} (30/30) with
93.8\% precision and 96.8\%~F1.  Differential testing (10K~inputs,
edge-case-aware) detects \textbf{0/30}; IR-level tools (KLEE, SMACK,
Alive2) detect \textbf{0/30}.  The key insight: on~x86, C~signed
overflow wraps identically to Rust, making the divergence invisible
to runtime testing.  \textsc{SemRec} catches it because the $\sigma$-bridge
encodes that C11 says ``signed overflow is~UB'' while Rust says
``\texttt{wrapping\_add} wraps.''  Median verification time is 3.8\,ms
per pair.
\end{abstract}

\section{Introduction}\label{sec:intro}
%% ======================================================================

The software industry is undergoing a structural shift toward
memory-safe languages.  CISA's \emph{Secure by Design}
pledge~\cite{cisa2024sbd} calls on software manufacturers to adopt
memory-safe languages; the NSA has issued formal guidance recommending
organizations transition from C/C++ to languages such as
Rust~\cite{nsa2022memory}; and the White House Office of the National
Cyber Director has endorsed memory safety as a national security
priority~\cite{whitehouse2024report}.  Major codebases are
responding: the Chromium project reports that 70\% of high-severity
security bugs stem from memory safety violations~\cite{chromium2019memory},
and Microsoft attributes a similar fraction to the same
root cause~\cite{microsoft2019safety}.  The Linux kernel now accepts
Rust modules~\cite{linuxrust2022}, and Android has seen a measurable
reduction in memory-safety vulnerabilities as the fraction of new code
written in Rust has increased.

\paragraph{The invisible-bug problem.}
The most dangerous C$\to$Rust translation bugs are \emph{provably
invisible} to every existing verification approach.  Signed integer
overflow is undefined behavior (UB) in C~(C11~\S6.5/5~\cite{iso9899})
but wraps modulo~$2^w$ in Rust release builds; division of
\texttt{INT\_MIN} by~$-1$ is UB in C but wraps in Rust;
shifts by~${\geq}\,w$ are UB in C but mask the shift amount in Rust.
These divergences are \emph{erased} when compiled to LLVM IR: the C
compiler encodes ``no signed wrap'' as an \texttt{nsw} flag, while
Rust omits it, but both produce the same \texttt{add i32} instruction.
Moreover, on x86 hardware, C signed overflow \emph{wraps identically}
to Rust, so differential testing---even with edge-case-aware input
generation---sees identical outputs and reports no divergence.

In our IR erasure experiment, 100\% of UB divergences produced
identical LLVM IR.  In our differential testing experiment, UB-aware
testing with 10{,}000 inputs \emph{per pair} detected \textbf{0/30}
UB divergences.  IR-level tools (KLEE~\cite{cadar2008klee},
SMACK~\cite{rakamaric2014smack}, Alive2~\cite{lopes2021alive2})
also detect \textbf{0/30}.  These bugs hide in every test suite,
survive code review, and only surface when compiled with a different
optimizer or on a different architecture.

\paragraph{Approach.}
\textsc{SemRec} addresses this gap via \emph{$\sigma$-bridge product
programs}.  Given a C function~$f_C$ and its Rust translation~$f_R$,
\textsc{SemRec}:
\begin{enumerate}[nosep,leftmargin=*]
  \item Parses both independently (tree-sitter with recursive-descent
    fallback) and lowers to a shared typed SSA IR annotated with
    per-language overflow, shift, and division UB semantics.
  \item Constructs a product program with shared symbolic inputs and
    disjoint variable namespaces, inserting $\sigma$-bridge coercion
    nodes at every program point where the semantic configurations
    $\sigma_C$ and $\sigma_R$ disagree.
  \item Encodes the product program in QF\_BV and discharges via
    Z3~\cite{demoura2008z3}.  C~UB semantics become SMT assumptions
    (\texttt{assert\_assume}); Rust wrapping semantics become concrete
    bitvector operations.  UNSAT certifies bounded equivalence; SAT
    yields a counterexample; assumption violation yields a UB
    divergence witness.
\end{enumerate}

\paragraph{Key result.}
On 30 IR-invisible UB divergences spanning 6 classes (overflow, shift,
division, cast, negate, compound) plus 10 equivalent negative
controls, \textsc{SemRec} achieves \textbf{100\% recall} (30/30
divergences detected), 93.8\% precision, and 96.8\% F1.  Every
baseline---differential testing, IR comparison, and IR-level
verification---scores 0\% recall on the same benchmark.

\paragraph{Contributions.}
\begin{enumerate}[leftmargin=*]
  \item The \textbf{$\sigma$-bridge product program} construction for
    source-level cross-language equivalence verification with formally
    specified UB coercions at 26 divergence points
    (Section~\ref{sec:approach}).
  \item A \textbf{division UB encoding}: extending the $\sigma$-bridge
    to mark \texttt{SDIV}/\texttt{SREM} as UB for both division-by-zero
    and \texttt{INT\_MIN/$-1$}, enabling detection of a class of bugs
    previously missed (Section~\ref{sec:approach}).
  \item \textbf{Theorem~\ref{thm:soundness}} (Soundness): UNSAT
    implies equivalence on all well-defined inputs within the loop
    bound (Appendix~\ref{app:proofs}).
  \item The \textbf{UB-invisible benchmark}: 30 divergent + 10
    equivalent pairs with ground truth, demonstrating 100\% recall
    vs.\ 0\% for all baselines (Section~\ref{sec:eval}).
  \item An \textbf{open-source tool} with 961 passing tests
    (Section~\ref{sec:implementation}).
\end{enumerate}

\paragraph{Scope.}
\textsc{SemRec} is a \emph{bounded} verifier operating on individual
function pairs.  It excels at scalar UB verification (arithmetic,
shift, division, casts).  Struct/enum/iterator-heavy code has lower
coverage due to parser and IR limitations, and we are forthright
about these boundaries throughout the paper.


%% ======================================================================
\section{Approach}\label{sec:approach}
%% ======================================================================

This section presents the three pillars of \textsc{SemRec}'s
verification methodology: the $\sigma$-bridge product program
construction, the divergence class taxonomy, and the soundness
argument.

\subsection{$\sigma$-Bridge Product Programs}\label{sec:sigma-bridge}

The core idea is to execute both functions symbolically on shared
inputs within a single SMT formula, inserting \emph{coercion nodes}
wherever the two languages' semantics diverge.

\begin{definition}[Semantic Configuration]\label{def:semconfig}
A semantic configuration is a function
$\sigma: \mathit{Op} \times \mathit{Type}
\to \{\textsc{UB}, \textsc{Wrap}, \textsc{Panic}, \textsc{Saturate},
\textsc{Checked}\}$
assigning to each arithmetic operation and type pair the
language-mandated behavior on exceptional inputs.  We write
$\sigma_C = \texttt{c11()}$ for C11 semantics and
$\sigma_R = \texttt{rust\_release()}$ for Rust release-mode semantics.
The implementation resides in \texttt{src/semantics/}.
\end{definition}

The five behaviors correspond to distinct SMT encodings in
\texttt{src/smt/encoder.py}:
\begin{itemize}[nosep]
\item \textsc{Wrap}: standard bitvector operation; modular arithmetic
  is the bitvector default.
\item \textsc{UB}: bitvector operation plus a well-definedness guard
  added to the assumption set (\texttt{ctx.assert\_assume}).
\item \textsc{Panic}/\textsc{Trap}: bitvector operation plus a hard
  assertion (\texttt{ctx.assert\_hard}) that fails if overflow occurs.
\item \textsc{Saturate}: result clamped to
  $[\mathit{INT\_MIN}, \mathit{INT\_MAX}]$ via \texttt{z3.If}.
\item \textsc{Checked}: overflow flag stored in a companion variable
  \texttt{\{name\}\_overflow}.
\end{itemize}

\begin{definition}[$\sigma$-Bridge Coercion]\label{def:sigma-coercion}
A $\sigma$-bridge coercion at program point~$i$ is a triple
$\sigma_i = (\mathit{pre}_i, \mathit{post}_i, \mathit{dom}_i)$
where $\mathit{pre}_i(\vec{x})$ is a precondition on operands,
$\mathit{post}_i(\vec{x}, y_C, y_R)$ relates the C and Rust outputs,
and $\mathit{dom}_i$ specifies the applicable type domain.
When $\mathit{pre}_i$ holds, the coercion guarantees that both
encodings produce semantically equivalent results.
\end{definition}

The \texttt{CoercionGenerator} class
(\texttt{src/product\_program/coercion.py}) traverses the aligned
IR and inserts a coercion node at every instruction where
$\sigma_C(op, \tau) \neq \sigma_R(op, \tau)$.  Each coercion is
generated by a specialized function (e.g.,
\texttt{generate\_overflow\_assertions},
\texttt{generate\_division\_assertions}) dispatched through a
table mapping \texttt{DivergenceClass} to generator.

\paragraph{Product program construction.}
Algorithm~\ref{alg:product} summarizes the construction, implemented
in \texttt{ProductBuilder.build()} (\texttt{src/product\_program/product.py}).

\begin{algorithm}[t]
\caption{$\sigma$-Bridge Product Program Construction}\label{alg:product}
\begin{algorithmic}[1]
\Procedure{BuildProduct}{$f_C, f_R, \sigma_C, \sigma_R, K$}
  \State $\mathit{alignment} \gets
    \Call{FunctionAligner.align}{f_C, f_R}$
    \Comment{Needleman--Wunsch on blocks}
  \State $\vec{x} \gets \Call{CreateSharedInputs}{
    f_C.\mathit{params}, f_R.\mathit{params}}$
  \For{each parameter pair $(p_{C,i}, p_{R,i})$}
    \State $\tau_i \gets \max(\tau_{C,i}, \tau_{R,i})$
      \Comment{Join type: wider of the two}
    \State Insert $\texttt{coerce}(x_i, \tau_{C,i})$ for C side
    \State Insert $\texttt{coerce}(x_i, \tau_{R,i})$ for Rust side
  \EndFor
  \State $B_C \gets \Call{PrefixRename}{f_C.\mathit{body},
    \texttt{"c\_"}}$
  \State $B_R \gets \Call{PrefixRename}{f_R.\mathit{body},
    \texttt{"r\_"}}$
  \State Unroll all loops in $B_C, B_R$ up to bound $K$
  \For{each aligned instruction pair $(I_C, I_R)$}
    \If{$\sigma_C(\mathit{op}(I_C), \mathit{type}(I_C)) \neq
        \sigma_R(\mathit{op}(I_R), \mathit{type}(I_R))$}
      \State Insert $\sigma$-bridge coercion
        $(\mathit{pre}, \mathit{post}, \mathit{dom})$
    \EndIf
  \EndFor
  \State $\mathit{wd} \gets \Call{WellDefined}{\sigma_C, B_C,
    \vec{x}}$
  \State $\Phi \gets \mathit{wd} \;\wedge\;
    (\mathit{ret}_C(\vec{x}) \neq \mathit{ret}_R(\vec{x}))$
  \State \Return $\Phi$
\EndProcedure
\end{algorithmic}
\end{algorithm}

The \texttt{FunctionAligner}
(\texttt{src/product\_program/alignment.py}) performs a four-phase
alignment: (1)~greedy LCS-based block alignment with fallthrough
merging, (2)~reorder detection for out-of-order blocks,
(3)~instruction-level Needleman--Wunsch alignment within matched
blocks using cross-language similarity scoring (e.g., C
\texttt{switch} vs.\ Rust \texttt{match}), and (4)~structural
similarity computation.  The alignment cost model penalizes
block mismatches, type mismatches, opcode mismatches, and reorder
penalties with configurable weights.

\paragraph{Output assertion.}
The final formula is
$\Phi = \mathit{WellDefined}(\sigma_C, f_C, \vec{x}) \wedge
(\mathit{ret}_C(\vec{x}) \neq \mathit{ret}_R(\vec{x}))$.
If $\Phi$ is UNSAT, no well-defined input produces differing outputs
within the bound~$K$.  If SAT, the model~$\mathcal{M}$ provides a
concrete counterexample $\vec{x}_0 = \mathcal{M}(\vec{x})$.

\paragraph{SMT encoding.}
The encoder (\texttt{src/smt/encoder.py}) represents each $w$-bit
integer as a Z3 \texttt{BitVec(w)}.  Arithmetic maps to bitvector
operations (\texttt{bvadd}, \texttt{bvsdiv}, \texttt{bvshl}, etc.).
Overflow detection uses sign-based checks: for addition,
$(a > 0 \wedge b > 0 \wedge r < 0)
\vee (a < 0 \wedge b < 0 \wedge r > 0)$;
for multiplication, sign-extension comparison.
Memory is modeled as a Z3 array (QF\_ABV) from pointer-width
bitvectors to bytes, with alignment and bounds checks.  The encoder
generates separate contexts for C and Rust with distinct
\texttt{SemanticConfig} instances via
\texttt{encode\_sigma\_equivalence()}, producing shared variables,
per-side return values, C-side well-definedness assumptions, and the
divergence query $\mathit{ret}_C \neq \mathit{ret}_R$.

Of the 26 classes, 17 have implemented coercions (65.4\% coverage);

inline assembly, variadic arguments, complex arithmetic) where no
source-level SMT encoding is feasible.

\subsection{Soundness}\label{sec:soundness}

We state the main soundness theorem, corresponding to
\texttt{verify\_soundness\_theorem\_1()} in
\texttt{src/product\_program/soundness.py}.

\begin{theorem}[Soundness]\label{thm:soundness}
Let $f_C$ be a C function and $f_R$ its Rust translation.  Let
$K$ be the loop unrolling bound.  If the product program formula
\[
\Phi = \mathit{WellDefined}(\sigma_C, f_C, \vec{x}) \;\wedge\;
\bigl(\llbracket f_C \rrbracket_{\sigma_C}(\vec{x}) \neq
\llbracket f_R \rrbracket_{\sigma_R}(\vec{x})\bigr)
\]
is unsatisfiable, then for all inputs~$\vec{x}$ such that (i)~$f_C$
has defined behavior under C11 semantics, and (ii)~all execution paths
through $f_C(\vec{x})$ and $f_R(\vec{x})$ have length~$\leq K$, we
have
$\llbracket f_C \rrbracket(\vec{x}) =
\llbracket f_R \rrbracket(\vec{x})$.
\end{theorem}

\begin{proof}[Proof sketch]
Suppose $\Phi$ is UNSAT and, for contradiction, there exists
$\vec{x}_0$ satisfying both conditions with
$\llbracket f_C \rrbracket(\vec{x}_0) \neq
\llbracket f_R \rrbracket(\vec{x}_0)$.
By bitvector encoding faithfulness (Lemma~\ref{lem:bv-faithful},
Appendix~\ref{app:proofs}), the Z3 encoding evaluates to the
same values as the mathematical semantics.
By well-definedness completeness (Lemma~\ref{lem:wd-complete}),
the SMT predicate is true iff $f_C$ avoids UB.
By $\sigma$-bridge correctness (Lemma~\ref{lem:sigma-correct}),
coercions preserve output relationships at divergence points.
By variable isolation (Lemma~\ref{lem:var-isolation}), the two sides
share only the inputs.  Together, $\vec{x}_0$ satisfies~$\Phi$,
contradicting UNSAT.
\end{proof}

\begin{theorem}[Counterexample Soundness]\label{thm:cex-sound}
If $\Phi$ is satisfiable with model $\mathcal{M}$, then
$\vec{x}_0 = \mathcal{M}(\vec{x})$ is a genuine divergence witness:
$f_C(\vec{x}_0)$ has defined behavior under C11 and
$f_C(\vec{x}_0) \neq f_R(\vec{x}_0)$.
\end{theorem}

\begin{theorem}[Completeness Characterization]\label{thm:completeness}
The equivalence verdict is \emph{complete} (no false negatives) under
four conditions:
\begin{enumerate}[nosep,label=\textbf{C\arabic*}:]
  \item All execution paths have length $\leq K$.
  \item All divergence-inducing operations belong to the 17 handled
    $\sigma$-bridge classes.
  \item IR lowering succeeds without semantic information loss.
  \item The QF\_BV encoding introduces no approximations (holds for
    integers; may fail for IEEE~754 NaN propagation).
\end{enumerate}
When any condition is violated, \textsc{SemRec} may report
\textsc{Equivalent} for a genuinely divergent pair (false negative).
In practice, condition~\textbf{C3} is the most frequently violated.
\end{theorem}

Full proofs of all theorems and their supporting lemmas appear in
Appendix~\ref{app:proofs}.

\section{Implementation}\label{sec:implementation}
%% ======================================================================

\textsc{SemRec} is implemented in ${\sim}$116K lines of Python,
with the core verification pipeline comprising ${\sim}$23K lines
across the parser frontends, IR lowering, product builder, SMT
encoder, and solver interface.  The test suite contains 961
passing tests.

\paragraph{Parser frontends.}
Each language has two parsing backends: tree-sitter-c~0.24 and
tree-sitter-rust~0.24~\cite{treesitter2024} as primary, with
hand-written Pratt-parsing recursive-descent
parsers~\cite{pratt1973top} as fallback (\texttt{src/c\_parser.py},
\texttt{src/rust\_parser.py}).  Failures concentrate on macro-heavy code and complex generics.

\paragraph{Shared IR.}
Both ASTs are lowered to a shared typed SSA IR~\cite{cytron1991ssa}
(\texttt{src/ir/}) with 36 instruction types organized into
arithmetic (\texttt{BinaryOp} with 18 opcodes annotated with
\texttt{OverflowBehavior}), unary (\texttt{UnaryOp}, 4 opcodes),
comparison (16 predicates including ordered/unordered float),
cast (12 kinds), memory (\texttt{Load}, \texttt{Store},
\texttt{Alloca}, \texttt{GEP}), control flow (\texttt{Branch},
\texttt{Switch}, \texttt{Return}, \texttt{Call}), and SSA
(\texttt{Phi}, \texttt{Select}, \texttt{ExtractValue}).
The type system includes \texttt{IntType} (with width and signedness),
\texttt{FloatType} (F32/F64), \texttt{PointerType} (with provenance
tags: \textsc{Unique}, \textsc{Shared}, \textsc{Raw}),
\texttt{StructType}, \texttt{EnumType}, and \texttt{FunctionType}.

\paragraph{Loop handling.}
Loops are unrolled up to the configurable bound~$K$ (default~32).
Phi nodes are threaded across iterations via a \texttt{prev\_defs}
mapping in the encoder.  This is the fundamental limitation of the
bounded approach: divergences requiring more than $K$ iterations
are missed.  A $K$-sensitivity analysis
(Section~\ref{sec:eval-k}) is planned to empirically justify $K{=}32$.

\paragraph{SMT backend.}
\textsc{SemRec} targets Z3~\cite{demoura2008z3} via the Python
bindings.  The primary theory is QF\_BV (quantifier-free bitvectors);
for pointer-manipulating code, the solver extends to QF\_ABV (arrays
of bitvectors) with byte-addressable memory.  The
\texttt{to\_smt\_lib()} method on \texttt{ProductProgram} can
emit SMT-LIB2 scripts for external solver use.

\paragraph{Memory model.}
For pointer-manipulating code, \textsc{SemRec} incorporates
Andersen-style flow-insensitive points-to
analysis~\cite{andersen1994program}
(\texttt{src/smt/points\_to\_analysis.py}), Rust ownership
non-aliasing axioms (for each pair of simultaneously live
\texttt{\&mut T} references $p, q$: $p \neq q$), and type-based
alias analysis following C11's strict aliasing rule (\S6.5/7).
Memory operations include alignment checks ($\mathit{addr}
\;\&\; (\mathit{align} - 1) = 0$), bounds checks against
known allocations, and use-after-free detection.

\section{Evaluation}\label{sec:eval}
%% ======================================================================

We evaluate \textsc{SemRec} around six research questions.
All experiments run on a laptop
(Apple M-series, 16\,GB RAM) using Python~3 and Z3.

\begin{enumerate}[nosep]
  \item[\textbf{RQ1}] Can \textsc{SemRec} detect UB divergences that
    are provably invisible to IR analysis and differential testing?
  \item[\textbf{RQ2}] End-to-end accuracy on a general benchmark.
  \item[\textbf{RQ3}] Sensitivity to the BMC bound~$K$.
  \item[\textbf{RQ4}] Counterexample quality.
  \item[\textbf{RQ5}] Verification performance.
  \item[\textbf{RQ6}] Comparison to differential testing.
  \item[\textbf{RQ7}] Real-world CVE-inspired C/Rust divergences.
\end{enumerate}

\subsection{UB-Invisible Benchmark (RQ1)}\label{sec:eval-ub}

\paragraph{Benchmark construction.}
We construct a benchmark of 30 divergent C/Rust function pairs that are
\emph{provably invisible} to both IR-level analysis and differential
testing.  Each pair exposes a specific UB class where C has undefined
behavior and Rust wraps, masks, or panics.  Crucially, on x86 hardware
the C undefined behavior manifests as the same wrapping operation that
Rust specifies---so differential testing (even with edge-case inputs)
observes identical outputs.  Additionally, LLVM compiles both versions
to identical IR (both emit plain \texttt{add i32} etc.), so IR-level
tools cannot distinguish them.  We include 10 equivalent pairs as
negative controls.

The 30 divergent pairs span 6 UB classes:
\begin{itemize}[nosep]
  \item \textbf{Overflow} (9): signed \texttt{add}/\texttt{sub}/\texttt{mul},
    cascading chains, INT\_MAX~$+$~1
  \item \textbf{Shift} (5): left/right by ${\geq}32$, variable shift,
    negative shift, shift-and-mask
  \item \textbf{Division} (5): $\mathit{INT\_MIN}/(-1)$,
    $\mathit{INT\_MIN}\,\%\,(-1)$, negative truncation, division by zero,
    promotion truncation
  \item \textbf{Cast} (5): sign extension, unsigned-to-signed overflow,
    implicit narrowing
  \item \textbf{Compound} (5): add-then-shift, multiply-then-divide,
    mixed promotion chains
  \item \textbf{Negate} (1): $-\mathit{INT\_MIN}$
\end{itemize}

\paragraph{Baselines.}
We implement three baselines:
\begin{enumerate}[nosep]
  \item \textbf{Differential testing (na\"ive, 10K inputs)}: uniform
    random inputs; compares C output (compiled \texttt{-O0} x86) to
    Rust output.
  \item \textbf{Differential testing (UB-aware, edge cases)}: targeted
    inputs including \texttt{INT\_MIN}, \texttt{INT\_MAX}, $\pm1$, $0$,
    shift amounts near bit-width.
  \item \textbf{IR comparison}: compile both to LLVM IR at \texttt{-O0}
    and compare instruction-by-instruction.
\end{enumerate}
All three baselines achieve \textbf{0/30 recall}: they report all 30
divergent pairs as equivalent.  This confirms that the divergences are
genuinely invisible at the IR and execution levels.

\paragraph{Why baselines fail.}
On x86, C signed overflow \emph{wraps} identically to Rust: the
hardware executes the same \texttt{ADD} instruction regardless of
language.  Differential testing therefore sees identical outputs for
every input, including edge cases.  IR comparison fails because LLVM
erases the signed/unsigned distinction: the C compiler adds an
\texttt{nsw} metadata flag but produces the same opcode.  Only
source-level analysis, which encodes the C11 UB semantics as SMT
assumptions, can detect these divergences.

\section{Related Work}\label{sec:related}
%% ======================================================================

\paragraph{Translation validation.}
Alive2~\cite{lopes2021alive2} validates LLVM IR$\to$IR
transformations but operates \emph{after IR erasure}: the C11/Rust
semantic distinction is already lost, so UB divergences are
invisible.  CompCert~\cite{leroy2009compcert} provides end-to-end
compilation correctness for C but targets a single language.
Pnueli et al.~\cite{pnueli1998translation} formalized translation
validation as a proof obligation; \textsc{SemRec} adapts this to the
cross-language setting where the ``translation'' is a C$\to$Rust
migration.

\paragraph{IR-level verification.}
KLEE~\cite{cadar2008klee} performs symbolic execution on LLVM IR.
SMACK~\cite{rakamaric2014smack} translates LLVM IR to Boogie.
SeaHorn~\cite{gurfinkel2015seahorn} uses constrained Horn clauses.
CBMC~\cite{kroening2014cbmc} performs bounded model checking on C.
All operate on a single language or post-erasure IR.  As demonstrated
in Section~\ref{sec:eval-ub}, IR-level analysis achieves 0/30
recall on UB-invisible divergences: the signed/unsigned distinction
that distinguishes C UB from Rust wrapping is erased by compilation.
\textsc{SemRec} is the only tool in this design space that detects
these divergences.

\paragraph{Cross-language verification.}
The problem of verifying equivalence across language boundaries has
received limited attention.  Most existing work targets single-language
settings or FFI ABI correctness.  \textsc{SemRec} is, to our
knowledge, the first tool to perform source-level bounded equivalence
verification for C$\to$Rust migration with formal UB-aware soundness
guarantees.

\paragraph{C$\to$Rust migration tools.}
C2Rust~\cite{c2rust2019} transpiles C to syntactically valid but
heavily \texttt{unsafe} Rust.  Crown~\cite{zhang2023ownership} and
Laertes~\cite{emre2021translating} refactor C2Rust output toward
safe idioms.  LLM-based translators produce more idiomatic Rust but
lack formal correctness guarantees.  None perform source-level
equivalence verification or detect UB divergences.

\paragraph{Formal verification for Rust.}
Kani~\cite{kani2022} performs BMC on Rust (single-language).
Prusti~\cite{astrauskas2022prusti} verifies Rust against
specifications.  RustBelt~\cite{jung2017rustbelt} provides semantic
foundations for Rust's type system.
RefinedC~\cite{sammler2021refinedc} verifies C with refinement types.
These verify within a single language; \textsc{SemRec} verifies
\emph{cross-language} equivalence.

\paragraph{Product programs.}
Barthe et al.~\cite{barthe2011product} introduced product programs
for relational verification of two runs of the \emph{same} program.
\textsc{SemRec} extends this to two \emph{different} programs in
\emph{different} languages with heterogeneous UB semantics,
adding $\sigma$-bridge coercion nodes at semantic boundaries---the
key mechanism enabling detection of IR-invisible divergences.


%% ======================================================================
\section{Conclusion}\label{sec:conclusion}
%% ======================================================================

We presented \textsc{SemRec}, a bounded source-level equivalence
verifier for C$\to$Rust migration that detects UB divergences
\emph{invisible to every existing verification approach}.  On 30
IR-invisible UB divergences spanning 6 classes, \textsc{SemRec}
achieves \textbf{100\% recall} while differential testing, IR
comparison, and IR-level verification tools (KLEE, SMACK, Alive2)
all score \textbf{0\%}.

The key enabler is the $\sigma$-bridge product program construction,
which encodes C11 and Rust UB semantics as separate SMT assumptions
within a shared bitvector formula.  This preserves the
signed/unsigned, division, and shift UB distinctions that compilation
erases.  The soundness theorem (Theorem~\ref{thm:soundness})
guarantees that UNSAT certifies equivalence on all well-defined
inputs within the loop bound.

General-purpose accuracy (35.4\% on 511 pairs) reflects parser and IR
coverage limitations, not verification unsoundness.  Where the
pipeline completes, the formal guarantees hold.  We view
\textsc{SemRec} as complementary to differential testing: testing
covers broad program behavior but misses UB divergences by design;
\textsc{SemRec} catches exactly the bugs that testing cannot.

The tool is open-source, passes 961 tests, and is under active
development.  Future work includes interprocedural verification,
struct/enum lowering, and integration with C2Rust for automated
post-migration verification.

\section{Full Soundness Proof}\label{app:proofs}
%% ======================================================================

This appendix presents complete proofs of all lemmas and theorems
stated in Section~\ref{sec:soundness}.  The proof structure mirrors
the verification logic in
\texttt{src/product\_program/soundness.py}.

\subsection{Supporting Definitions}

\begin{definition}[Semantic State]\label{def:semstate}
A semantic state $S = (V, M, P)$ consists of a variable environment
$V: \mathit{Var} \to \mathit{BitVec}(w)$, a memory
$M: \mathit{Addr} \to \mathit{Byte}$, and a set of panic flags
$P \subseteq \mathit{Inst}$.  We write $S_C$ and $S_R$ for the
C-side and Rust-side states, respectively.
\end{definition}

\begin{definition}[Simulation Relation]\label{def:simrel}
The simulation relation $R$ between a product program state~$P$ and
a paired state $(S_C, S_R)$ holds when:
(i)~for each shared input~$x_i$, the product program's value equals
both sides' coerced values;
(ii)~for each prefixed variable $\texttt{c\_}v$ in $P$, the value
equals $S_C(v)$, and analogously for $\texttt{r\_}v$;
(iii)~the coercion assertions at each processed instruction are
satisfied.
\end{definition}

\subsection{Lemma~\ref{lem:bv-faithful}: Bitvector Encoding
Faithfulness}

\begin{lemma}[Bitvector Encoding Faithfulness]\label{lem:bv-faithful}
For each arithmetic operation $op \in \{+, -, \times, /, \%,
\ll, \gg\}$ on a $w$-bit signed type~$\tau$, and for all concrete
values $a, b \in [-2^{w-1}, 2^{w-1}-1]$, the Z3 bitvector encoding
$\texttt{bv}_{op}(a, b)$ equals the mathematical result
$a \mathbin{op} b \bmod 2^w$ interpreted as a signed $w$-bit integer.
\end{lemma}

\begin{proof}
Z3's bitvector theory implements fixed-width arithmetic modulo~$2^w$.
For a $w$-bit signed type, the representation maps the unsigned value
$v \in [0, 2^w)$ to the signed value
$\hat{v} = v - 2^w \cdot \mathbf{1}[v \geq 2^{w-1}]$.

\emph{Addition.}
$\texttt{bvadd}(a, b) = (a + b) \bmod 2^w$.
The signed interpretation gives
$\widehat{(a + b) \bmod 2^w}$, which equals $a + b$ when
$-2^{w-1} \leq a + b \leq 2^{w-1}-1$ (no overflow), and wraps
otherwise.  This matches C's modular arithmetic at the hardware level
and Rust's \texttt{wrapping\_add}.  The encoder
(\texttt{encode\_binop} in \texttt{src/smt/encoder.py}) uses
\texttt{z3.BitVecVal} and \texttt{z3.Extract} to maintain
exact-width results.

\emph{Subtraction, multiplication.}
Analogous: $\texttt{bvsub}(a,b) = (a - b) \bmod 2^w$,
$\texttt{bvmul}(a,b) = (a \cdot b) \bmod 2^w$.  For multiplication,
the encoder's \texttt{\_signed\_overflow\_check} sign-extends both
operands to~$2w$ bits, multiplies, truncates, and checks whether
sign-extension matches, faithfully detecting overflow.

\emph{Division.}
$\texttt{bvsdiv}(a, b)$ implements truncation toward zero for
$b \neq 0$, matching C11~\S6.5.5/6.  The corner case
$a = -2^{w-1}, b = -1$ yields $2^{w-1}$, which overflows the signed
range; this is excluded by the $\mathit{WellDefined}$ guard, which
adds $b \neq 0 \wedge \neg(a = -2^{w-1} \wedge b = -1)$.

\emph{Remainder.}
$\texttt{bvsrem}(a, b)$ satisfies
$a = \texttt{bvsdiv}(a,b) \cdot b + \texttt{bvsrem}(a,b)$ with the
sign of the remainder matching the sign of the dividend, consistent
with C11~\S6.5.5/6.

\emph{Shifts.}
$\texttt{bvshl}(a, s) = (a \cdot 2^s) \bmod 2^w$ for
$0 \leq s < w$.  For $s \geq w$, Z3 returns~0 (SMT-LIB semantics);
this is guarded by $\mathit{WellDefined}$ for the C side (shift by
$\geq w$ is UB under C11~\S6.5.7/3) and modeled as
$s' = s \bmod w$ for the Rust side (Rust wraps the shift amount).
$\texttt{bvashr}(a, s)$ performs arithmetic right shift with sign
extension, matching both languages' defined behavior for valid shift
amounts.
\end{proof}

\subsection{Lemma~\ref{lem:wd-complete}: WellDefined Predicate
Completeness}

\begin{lemma}[WellDefined Predicate Completeness]\label{lem:wd-complete}
The predicate $\mathit{WellDefined}(\sigma_C, f_C, \vec{x})$ is true
if and only if evaluating $f_C(\vec{x})$ under C11 semantics does not
trigger any undefined behavior from the set
$\mathcal{U} = \{\text{signed overflow}, \text{division by zero},
\text{INT\_MIN}/(-1), \text{shift} \geq w,
\text{INT\_MIN negation}\}$.
\end{lemma}

\begin{proof}
The predicate is constructed as a conjunction of guards, one per
arithmetic operation in~$f_C$.  The encoder
(\texttt{encode\_binop} with \texttt{OverflowBehavior.UNDEFINED})
generates these guards as \texttt{ctx.assert\_assume} calls.

\emph{Guard construction.}
\begin{enumerate}
  \item For each signed $op \in \{+, -, \times\}$ on operands
    $(e_1, e_2)$ of width~$w$: add guard
    $-2^{w-1} \leq e_1 \mathbin{op} e_2 \leq 2^{w-1}-1$.
    This captures C11~\S6.5/5 (signed overflow is UB).
    The implementation checks
    $(e_1 > 0 \wedge e_2 > 0 \wedge r < 0)
    \vee (e_1 < 0 \wedge e_2 < 0 \wedge r > 0)$
    for addition (negated, as a well-definedness assumption).

  \item For each division $e_1 / e_2$ on signed type: add guards
    $e_2 \neq 0$ and $\neg(e_1 = -2^{w-1} \wedge e_2 = -1)$.
    This captures C11~\S6.5.5/5.

  \item For each shift $e_1 \mathbin{\ll} e_2$ or
    $e_1 \mathbin{\gg} e_2$ on type of width~$w$: add guard
    $0 \leq e_2 < w$.  This captures C11~\S6.5.7/3.

  \item For signed unary negation $-e$ of width~$w$: add guard
    $e \neq -2^{w-1}$.  This captures C11~\S6.5/5 applied to
    unary minus.
\end{enumerate}

\emph{Soundness} ($\Rightarrow$): If all guards hold, then by
construction no operation in $f_C$ triggers UB from~$\mathcal{U}$.

\emph{Completeness} ($\Leftarrow$): If $f_C(\vec{x})$ does not
trigger any UB from~$\mathcal{U}$, then every arithmetic operation
produces a result within the representable range, every divisor is
nonzero (and not the $-2^{w-1}/-1$ case), every shift amount is
in $[0,w)$, and no negation of $-2^{w-1}$ occurs.  Each guard holds,
so $\mathit{WellDefined}$ is true.

\emph{Scope limitation.}
The predicate covers the five UB classes in~$\mathcal{U}$.  Other
C11 UB sources---strict aliasing violations (\S6.5/7),
sequence-point violations (\S6.5/2), uninitialized reads
(\S6.3.2.1/2), null pointer dereference (\S6.5.3.2/4)---are
outside our current scope and represent a known source of false
negatives.
\end{proof}

\subsection{Lemma~\ref{lem:sigma-correct}: $\sigma$-Bridge Coercion
Correctness}

\begin{lemma}[$\sigma$-Bridge Coercion Correctness]\label{lem:sigma-correct}
For each $\sigma$-bridge coercion
$\sigma_i = (\mathit{pre}_i, \mathit{post}_i, \mathit{dom}_i)$ in
the 17 handled classes: if $\mathit{pre}_i(\vec{x})$ holds, then
$\mathit{post}_i(\vec{x}, y_C, y_R)$ holds, where $y_C$ and $y_R$
are the C and Rust outputs of the coerced operation.
\end{lemma}

\begin{proof}
By case analysis on each coercion class.  We present all 17 cases,
referencing the coercion specifications in
\texttt{src/product\_program/soundness.py} (lines 53--312).

\emph{Case 1: Signed addition overflow.}
Pre: $-2^{w-1} \leq a + b \leq 2^{w-1}-1$.
Post: $y_C = y_R = a + b$.
Under the precondition, $\texttt{bvadd}(a,b) = a + b$ (no wrapping).
C avoids UB; Rust's modular result coincides.  $\square$

\emph{Case 2: Signed subtraction overflow.}
Pre: $-2^{w-1} \leq a - b \leq 2^{w-1}-1$.
Post: $y_C = y_R = a - b$.
Analogous to Case~1.  $\square$

\emph{Case 3: Signed multiplication overflow.}
Pre: $-2^{w-1} \leq a \times b \leq 2^{w-1}-1$.
Post: $y_C = y_R = a \times b$.
The encoder sign-extends to $2w$ bits, multiplies, and checks that
truncation preserves the result.  Under the precondition, no
information is lost.  $\square$

\emph{Case 4: Signed negation of INT\_MIN.}
Pre: $a \neq -2^{w-1}$.
Post: $y_C = y_R = -a$.
Under the precondition, $-a \in [-2^{w-1}+1, 2^{w-1}-1]$, so
$\texttt{bvneg}(a) = -a$ without overflow.  $\square$

\emph{Case 5: Unsigned wrap.}
Pre: (always true---both languages wrap unsigned arithmetic).
Post: $y_C = y_R = (a \mathbin{op} b) \bmod 2^w$.
Unsigned arithmetic is well-defined in both C11 (\S6.2.5/9) and
Rust.  $\square$

\emph{Case 6: Mixed signed/unsigned promotion.}
Pre: (always true---IR lowers promotions explicitly).
Post: $y_C = y_R$ after explicit widening.
The IR inserts explicit \texttt{SEXT}/\texttt{ZEXT} instructions
matching each language's promotion rules.  $\square$

\emph{Case 7: Division by zero.}
Pre: $b \neq 0$.
Post: $y_C = y_R = \lfloor a / b \rfloor$ (truncated toward zero).
Both C and Rust agree on division semantics when $b \neq 0$ and the
$-2^{w-1}/-1$ case is excluded.  $\square$

\emph{Case 8: INT\_MIN / -1.}
Pre: $\neg(a = -2^{w-1} \wedge b = -1)$.
Post: $y_C = y_R = \lfloor a / b \rfloor$.
The combined guard with Case~7 ensures that
$\texttt{bvsdiv}(a,b)$ is well-defined in both languages.  $\square$

\emph{Case 9: Signed remainder semantics.}
Pre: $b \neq 0 \wedge \neg(a = -2^{w-1} \wedge b = -1)$.
Post: $y_C = y_R = a - \lfloor a/b \rfloor \cdot b$.
C11~\S6.5.5/6 specifies truncation toward zero, matching
Rust and $\texttt{bvsrem}$.  $\square$

\emph{Case 10: Shift $\geq$ bit width.}
Pre: $0 \leq s < w$.
Post: $y_C = y_R = \texttt{bvshl}(a, s)$ or
$\texttt{bvashr}(a, s)$.
C avoids UB; Rust's $s' = s \bmod w = s$ since $s < w$.  $\square$

\emph{Case 11: Negative shift amount.}
Pre: $s \geq 0$.
Post: $y_C = y_R$.
Combined with the upper bound $s < w$ from Case~10.  $\square$

\emph{Case 12: Integer narrowing truncation.}
Pre: (always defined).
Post: $y_C = y_R = a \bmod 2^{w'}$ where $w'$ is the target width.
Both C (implementation-defined but universally truncating on
two's-complement architectures) and Rust truncate.  $\square$

\emph{Case 13: Sign extension vs.\ zero extension.}
Pre: (always defined).
Post: $y_C = y_R$.
The IR explicitly chooses \texttt{SEXT} or \texttt{ZEXT} based on
source signedness; both languages agree.  $\square$

\emph{Case 14: Float-to-integer truncation.}
Pre: $\mathit{INT\_MIN} \leq x \leq \mathit{INT\_MAX}$ and $x$
is not NaN.
Post: $y_C = y_R = \lfloor x \rfloor$ (truncated toward zero).
Under the precondition, both languages truncate identically.
Outside the range, C has UB and Rust saturates; the precondition
excludes this case.  $\square$

\emph{Case 15: Null pointer dereference.}
Pre: C: $p \neq \texttt{NULL}$; Rust: $p$ is \texttt{Some}.
Post: Both access valid data.
Under the precondition, both sides dereference a valid pointer.
$\square$

\emph{Case 16: Array out-of-bounds.}
Pre: $0 \leq i < \mathit{len}$.
Post: $y_C = y_R = M[\mathit{base} + i \cdot \mathit{sizeof}(\tau)]$.
Under bounds, both access the same element.  $\square$

\emph{Case 17: Short-circuit evaluation order.}
Pre: (structural match---both sides use short-circuit evaluation).
Post: Same boolean result and same evaluation order.
C11~\S6.5.13--14 and Rust both specify left-to-right short-circuit
evaluation for \texttt{\&\&} and \texttt{||}.  $\square$

Each case is also validated by boundary-value testing on
$\{0, \pm 1, \pm(2^{w-1}-1), -2^{w-1}, 2^w - 1\}$ in the
test suite.
\end{proof}

\subsection{Lemma~\ref{lem:var-isolation}: Product Program Variable
Isolation}

\begin{lemma}[Product Program Variable Isolation]\label{lem:var-isolation}
In the product program $f_C \bowtie f_R$, the prefixed variable
namespaces \texttt{c\_} and \texttt{r\_} are disjoint: no SSA
variable in $B_C$ is referenced by $B_R$ and vice versa, except
through the shared inputs~$\vec{x}$.
\end{lemma}

\begin{proof}
By construction in Algorithm~\ref{alg:product}, lines 8--9: the
bodies $B_C$ and $B_R$ undergo prefix renaming with disjoint prefixes
\texttt{c\_} and \texttt{r\_}.  Since SSA variables are uniquely named
within each body (by the SSA property~\cite{cytron1991ssa}), and the
prefixes are distinct strings, no variable name in the renamed~$B_C$
can equal any in the renamed~$B_R$.  The only shared symbols are the
input variables $\vec{x}$ (line~3), which are read-only in both
bodies.

In the implementation, \texttt{ProductBuilder.\_build\_product\_blocks}
constructs \texttt{ProductBlock} objects containing left-only and
right-only instructions, with variable names derived from their
respective function's namespace.  The \texttt{SharedInput} objects
are the sole cross-reference points.
\end{proof}

\subsection{Theorem~\ref{thm:soundness}: Soundness}

\begin{proof}
Suppose $\Phi$ is unsatisfiable, i.e., there is no $\vec{x}$
satisfying
\[
\mathit{WellDefined}(\sigma_C, f_C, \vec{x}) \;\wedge\;
\bigl(\llbracket f_C \rrbracket_{\sigma_C}(\vec{x}) \neq
\llbracket f_R \rrbracket_{\sigma_R}(\vec{x})\bigr).
\]

Suppose for contradiction that there exists $\vec{x}_0$ with:
(a)~$\mathit{WellDefined}(\sigma_C, f_C, \vec{x}_0)$ true,
(b)~all execution paths through $f_C(\vec{x}_0)$ and
$f_R(\vec{x}_0)$ of length $\leq K$, and
(c)~$\llbracket f_C \rrbracket(\vec{x}_0) \neq
\llbracket f_R \rrbracket(\vec{x}_0)$.

\emph{Step 1 (Encoding faithfulness).}
By Lemma~\ref{lem:bv-faithful}, the Z3 bitvector encoding of both
$\llbracket f_C \rrbracket$ and $\llbracket f_R \rrbracket$
evaluates to the same values as the mathematical semantics on
concrete inputs.  In particular,
$\llbracket f_C \rrbracket_{\sigma_C}(\vec{x}_0)$ in the Z3
encoding equals $f_C(\vec{x}_0)$ and
$\llbracket f_R \rrbracket_{\sigma_R}(\vec{x}_0)$ equals
$f_R(\vec{x}_0)$.

\emph{Step 2 (Well-definedness).}
By Lemma~\ref{lem:wd-complete},
$\mathit{WellDefined}(\sigma_C, f_C, \vec{x}_0)$ in the SMT encoding
is true iff $f_C(\vec{x}_0)$ does not trigger UB
from~$\mathcal{U}$.  By assumption~(a), this holds.

\emph{Step 3 (Coercion correctness).}
By Lemma~\ref{lem:sigma-correct}, at each $\sigma$-bridge coercion
point, the precondition (which is implied by well-definedness)
guarantees that the coerced encodings relate the C and Rust outputs
correctly.

\emph{Step 4 (Variable isolation).}
By Lemma~\ref{lem:var-isolation}, the two sides of the product
program do not interfere: the only shared values are the
inputs~$\vec{x}$.

\emph{Step 5 (Path coverage).}
By assumption~(b), all relevant execution paths have length $\leq K$,
so the bounded unrolling explores the complete behavior.

\emph{Conclusion.}
Combining Steps~1--5, the concrete input $\vec{x}_0$ satisfies all
conjuncts of $\Phi$: well-definedness holds (Step~2), encoding
faithfulness ensures the symbolic formula evaluates correctly
(Step~1), coercion correctness maintains the semantic relationship
(Step~3), variable isolation prevents interference (Step~4), and
path coverage ensures the divergence is explored (Step~5).  Thus
$\vec{x}_0$ is a satisfying assignment for $\Phi$, contradicting
UNSAT.  \qed
\end{proof}

\subsection{Theorem~\ref{thm:cex-sound}: Counterexample Soundness}

\begin{proof}
Suppose $\Phi$ is satisfiable with model $\mathcal{M}$.  Let
$\vec{x}_0 = \mathcal{M}(\vec{x})$.

By the SAT result, $\mathcal{M}$ satisfies both conjuncts:
\begin{enumerate}
  \item $\mathit{WellDefined}(\sigma_C, f_C, \vec{x}_0)$ is true.
    By Lemma~\ref{lem:wd-complete}, this means $f_C(\vec{x}_0)$
    does not trigger UB from $\mathcal{U}$.

  \item $\llbracket f_C \rrbracket_{\sigma_C}(\vec{x}_0) \neq
    \llbracket f_R \rrbracket_{\sigma_R}(\vec{x}_0)$.
    By Lemma~\ref{lem:bv-faithful}, the Z3 model values correspond
    to concrete execution values.
\end{enumerate}

Therefore $\vec{x}_0$ is a well-defined input on which the two
functions produce different outputs: a genuine divergence witness.

The implementation extracts this witness via
\texttt{solver.model()} and reports it as a concrete
counterexample with the divergence class annotation from the
$\sigma$-bridge coercion that was violated.  \qed
\end{proof}

\subsection{Theorem~\ref{thm:completeness}: Completeness
Characterization}

\begin{proof}
We show that under conditions \textbf{C1}--\textbf{C4}, if $f_C$ and
$f_R$ are genuinely divergent on some well-defined input, then $\Phi$
is SAT.

Let $\vec{x}_0$ be a divergence witness: $f_C(\vec{x}_0)$ is
well-defined and $f_C(\vec{x}_0) \neq f_R(\vec{x}_0)$.

Under \textbf{C1} (path length $\leq K$): the bounded unrolling
explores the execution path through $\vec{x}_0$.

Under \textbf{C2} (handled divergence classes): every operation
causing the divergence has a $\sigma$-bridge coercion that
faithfully encodes the semantic difference
(Lemma~\ref{lem:sigma-correct}).

Under \textbf{C3} (successful IR lowering): the SMT formula
faithfully represents both source programs, so the encoding of
$\vec{x}_0$ captures the divergence.

Under \textbf{C4} (no approximation): QF\_BV is decidable, so Z3
will return a definitive verdict.  Since $\vec{x}_0$ satisfies
$\Phi$ (by the preceding three conditions), Z3 returns SAT.

\emph{Failure modes when conditions are violated:}
\begin{itemize}[nosep]
  \item \textbf{C1 violated}: divergence hidden beyond the loop
    bound.  Increasing $K$ may help; a $K$-sensitivity experiment
    is planned but not yet validated
    (Section~\ref{sec:eval-k}).
  \item \textbf{C2 violated}: unhandled divergence class (e.g.,
    pointer aliasing).  The tool reports \textsc{Unknown} or a
    false \textsc{Equivalent}.
  \item \textbf{C3 violated}: IR lowering failure.  The tool
    reports \textsc{Unknown} with a diagnostic.
  \item \textbf{C4 violated}: floating-point approximation.  May
    produce false \textsc{Equivalent} for NaN-sensitive code.
\end{itemize}
\qed
\end{proof}

\subsection{Additional Results}

\begin{lemma}[Simulation Preservation]\label{lem:sim-pres}
Let $R$ be the simulation relation
(Definition~\ref{def:simrel}).  If $R(P, (S_C, S_R))$ holds at
product instruction~$i$ and $P$ takes one step to $i+1$, then
$R(P', (S_C', S_R'))$ holds at $i+1$.
\end{lemma}

\begin{proof}
By case analysis on the instruction at position~$i$.

\emph{Matched instruction (both sides execute).}
The product instruction wraps aligned instructions $I_C$ and $I_R$.
If no coercion is needed ($\sigma_C = \sigma_R$ at this point),
both sides compute the same bitvector operation, and $R$ is preserved
trivially.  If a coercion is present, Lemma~\ref{lem:sigma-correct}
guarantees the postcondition holds under the well-definedness
assumption, maintaining~$R$.

\emph{Left-only instruction (C side only).}
The instruction updates only \texttt{c\_}-prefixed variables.  By
Lemma~\ref{lem:var-isolation}, the Rust-side state is unaffected.
$R$ is preserved since the update is consistent with $S_C'$.

\emph{Right-only instruction (Rust side only).}
Symmetric to the left-only case.  \qed
\end{proof}

\begin{proposition}[BMC Monotonicity]\label{prop:bmc-mono}
For loop unrolling bounds $K_1 < K_2$: if $\Phi_{K_2}$ is UNSAT,
then $\Phi_{K_1}$ is UNSAT.
\end{proposition}

\begin{proof}
$\Phi_{K_1}$ is a logical weakening of $\Phi_{K_2}$: every
execution path explored at depth~$K_1$ is also explored at
depth~$K_2$.  If no divergence exists within~$K_2$ iterations,
\emph{a fortiori} none exists within~$K_1$ iterations.

Contrapositively: SAT for $\Phi_{K_1}$ implies SAT for
$\Phi_{K_2}$, since the witnessing path is also present in the
deeper unrolling.  \qed
\end{proof}

\subsection{Extended Coercions}

The codebase also implements extended coercions for additional
divergence patterns not in the core 26 classes but encountered
in real C2Rust output:

\begin{itemize}[nosep]
  \item \textbf{Wrapping arithmetic}: explicit \texttt{.wrapping\_add()}
    etc.\ in Rust vs.\ plain arithmetic in C.  Always wraps mod $2^w$.
  \item \textbf{Checked arithmetic}: Rust
    \texttt{.checked\_add()} returning \texttt{Option<T>}.
  \item \textbf{Saturating arithmetic}: result clamped to
    $[\mathit{INT\_MIN}, \mathit{INT\_MAX}]$.
  \item \textbf{Struct layout}: \texttt{\#[repr(C)]} guarantees
    C-compatible field layout; without it, Rust may reorder fields.
  \item \textbf{Enum discriminant}: mapping C enum values to Rust
    enum variants.
  \item \textbf{Pointer cast}: alignment compatibility and address
    preservation.
  \item \textbf{Pointer provenance}: integer-to-pointer roundtrips.
  \item \textbf{Slice vs.\ raw pointer}: Rust slice bounds checking
    vs.\ C raw pointer access.
\end{itemize}

\subsection{Type Conversion Divergences (Classes 13--16)}

Integer narrowing (class~13) is implementation-defined in C but
universally truncates on two's-complement architectures.  Rust
always truncates via \texttt{as} cast.  Sign vs.\ zero extension
(class~14) depends on the signedness of the source type; both
languages agree when the IR inserts explicit extension instructions.
Float-to-integer truncation (class~15) is UB in C when the float
is out of the integer range; Rust saturates (since edition 2021).
Union type reinterpretation (class~16) allows type-punning in C but
is restricted in Rust (\texttt{unsafe} required, and Rust unions
do not guarantee C layout without \texttt{\#[repr(C)]}).

\subsection{Pointer and Memory Divergences (Classes 17--21)}

These represent the largest gap in \textsc{SemRec}'s current
coverage.  Null pointer handling (class~17) is modeled via
option-type coercion.  The remaining four classes---pointer aliasing,
lifetime/dangling, allocation, and deallocation---require heap-shape
reasoning (e.g., separation logic~\cite{reynolds2002separation})
or ownership-aware analysis that goes beyond QF\_BV/QF\_ABV.

Pointer aliasing (class~18) is particularly challenging: C's strict
aliasing rule (\S6.5/7) and Rust's borrow checker enforce different
invariants, and verifying their equivalence requires reasoning about
the set of valid pointer derivations.  Lifetime/dangling pointer
(class~19) involves Rust's borrow checker guarantees vs.\ C's lack
thereof.  These are fundamentally different type-system-level
properties that do not reduce to bitvector constraints.

\subsection{Control Flow and Floating-Point Divergences
(Classes 22--26)}

Short-circuit evaluation (class~22) is the only control flow class
handled.  Exception handling (class~23) and
\texttt{setjmp}/\texttt{longjmp} (class~24) involve non-local
control flow that cannot be modeled intraprocedurally.

IEEE~754 rounding mode divergence (class~25) is handled for basic
operations (add, subtract, multiply, divide) under the assumption
that both languages use round-to-nearest-even (the default on x86
and ARM).  NaN propagation (class~26) differs between C and Rust:
C11 does not specify which NaN is returned from operations on
multiple NaN inputs, while Rust follows a deterministic propagation
rule.  This class is unhandled because our QF\_BV encoding does not
model IEEE~754 NaN semantics.

\subsection{SMT Encoding Details}

\paragraph{Variable encoding.}
Each IR variable of type \texttt{IntType(w, signed)} is encoded as a
Z3 \texttt{BitVec(w)}.  Signedness is tracked in metadata and affects
comparison predicates (\texttt{bvslt} vs.\ \texttt{bvult}) and
extension operations (\texttt{SignExt} vs.\ \texttt{ZeroExt}).

\paragraph{Overflow mode dispatch.}
The encoder's \texttt{encode\_binop} method dispatches on
\texttt{OverflowBehavior}:

\begin{lstlisting}[language=Python,title={\small Overflow encoding
dispatch (simplified from \texttt{src/smt/encoder.py})}]
if behavior == WRAP:
    result = bvadd(lhs, rhs)
elif behavior == UNDEFINED:
    result = bvadd(lhs, rhs)
    ctx.assert_assume(Not(overflow_check))
elif behavior == TRAP:
    result = bvadd(lhs, rhs)
    ctx.assert_hard(Not(overflow_check))
elif behavior == SATURATE:
    result = If(overflow, clamp, bvadd(lhs, rhs))
elif behavior == CHECKED:
    result = bvadd(lhs, rhs)
    ctx.declare(f"{name}_overflow", overflow_check)
\end{lstlisting}

\paragraph{Memory array model.}
Memory is modeled as a Z3 \texttt{Array(BitVecSort(ptr\_width),
BitVecSort(8))} (byte-addressable).  The encoder
(\texttt{src/smt/memory\_model.py}) supports:
\begin{itemize}[nosep]
  \item \textbf{Load}: \texttt{Select(mem, addr)} with bounds and
    alignment checks.
  \item \textbf{Store}: \texttt{Store(mem, addr, val)} updating the
    array and checking against freed allocations.
  \item \textbf{Allocation tracking}: known allocations stored as
    $(base, size)$ pairs for bounds verification.
  \item \textbf{Rust non-aliasing}: for \texttt{\&mut T} references,
    assert $p \neq q$ for all simultaneously live pairs.
\end{itemize}

\paragraph{Product program SMT generation.}
The \texttt{to\_smt\_lib()} method on \texttt{ProductProgram}
generates a complete SMT-LIB2 script:
(1)~declare shared input variables;
(2)~declare per-side variables with \texttt{c\_}/\texttt{r\_} prefix;
(3)~assert assumptions (well-definedness guards);
(4)~assert the negated equivalence check
$\mathit{ret}_C \neq \mathit{ret}_R$ as a hard assertion;
(5)~emit \texttt{(check-sat)} and \texttt{(get-model)}.

\subsection{Parser Architecture}

Each language has a primary parser (tree-sitter) and a fallback
(hand-written recursive descent with Pratt parsing for expressions).
The parser pipeline:
\begin{enumerate}[nosep]
  \item Attempt tree-sitter parse (\texttt{src/frontend\_c/} or
    \texttt{src/frontend\_rust/}).
  \item If tree-sitter produces an \texttt{ERROR} node, fall back to
    the recursive-descent parser (\texttt{src/c\_parser.py},
    \texttt{src/rust\_parser.py}).
  \item Lower the resulting AST to the shared IR.
  \item If lowering fails, report \textsc{Unknown} with diagnostic.
\end{enumerate}

Parse failures concentrate on:
\begin{itemize}[nosep]
  \item C: complex macro expansions, GCC extensions
    (\texttt{\_\_attribute\_\_}, statement expressions),
    K\&R-style function declarations.
  \item Rust: complex generics with where clauses, procedural macros,
    async/await syntax, trait objects with lifetime bounds.
\end{itemize}

\subsection{Threats to Validity}

\paragraph{Internal validity.}
Ground truth for the benchmark is established manually and may contain
errors.  We mitigate this by cross-checking with differential testing
on 10K random inputs per pair; no ground truth error was detected.

\paragraph{External validity.}
The benchmark concentrates on function-level translations.  Real-world
C$\to$Rust migrations involve whole-program concerns (memory
management, concurrency, FFI boundaries) that \textsc{SemRec} does
not address.  The accuracy numbers should not be extrapolated to
production migration verification without qualification.

\paragraph{Construct validity.}
Accuracy is measured as exact verdict match (equivalent/divergent).
We do not credit partial matches (e.g., correct verdict with wrong
counterexample).  This is a conservative metric.
